Step-by-step guide

Todo App with Supabase

Build a real-time todo list that works for anonymous users — no sign-up required. Uses Supabase Auth (anonymous sign-in), a Postgres table, and Row Level Security.

1

Create a Supabase project

Go to supabase.com and sign in. Click New project, choose an organization, give your project a name (e.g. todo-app), set a strong database password, and pick the region closest to your users. Click Create new project and wait ~1 minute for it to provision.

Once ready, head to Project Settings → API. You'll need two values:

  • Project URL — looks like https://xxxx.supabase.co
  • anon public key — safe to expose in frontend code
Tip: Keep the service_role key secret. For this app you only ever need the anon key on the client.
2

Enable anonymous sign-in

By default, Supabase requires an email or OAuth to create a session. Anonymous sign-in lets visitors get a real JWT without entering any credentials — perfect for a zero-friction todo app.

In your Supabase dashboard go to Authentication → Providers → Anonymous and toggle it on.

How it works: Each anonymous user gets a unique UUID stored in auth.users. Row Level Security will use that UUID to isolate each person's todos from everyone else's — with no login screen required.
3

Create the todos table

Open the SQL Editor in your dashboard and run the following. It creates the table, enables Row Level Security, and adds policies so each user can only see and modify their own rows.

SQL SQL Editor → New query 
CREATE TABLE todos (
  id         uuid        DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id    uuid        REFERENCES auth.users (id) ON DELETE CASCADE NOT NULL,
  task       text        NOT NULL,
  is_done    boolean     DEFAULT false NOT NULL,
  inserted_at timestamptz DEFAULT now() NOT NULL
);

-- Enable Row Level Security
ALTER TABLE todos ENABLE ROW LEVEL SECURITY;

-- Users can only read their own todos
CREATE POLICY "Users read own todos"
  ON todos FOR SELECT
  USING (auth.uid() = user_id);

-- Users can insert their own todos
CREATE POLICY "Users insert own todos"
  ON todos FOR INSERT
  WITH CHECK (auth.uid() = user_id);

-- Users can update their own todos
CREATE POLICY "Users update own todos"
  ON todos FOR UPDATE
  USING (auth.uid() = user_id);

-- Users can delete their own todos
CREATE POLICY "Users delete own todos"
  ON todos FOR DELETE
  USING (auth.uid() = user_id);
Why RLS? Without Row Level Security, any user with the anon key could read everyone's todos. These policies enforce per-user isolation at the database level — no server code needed.
4

Set up your project

Create a new folder for your app and install the Supabase JS client.

bash terminal 
mkdir supabase-todo && cd supabase-todo
npm init -y
npm install @supabase/supabase-js

Create a src/supabase.js file to initialise the client. Replace the placeholder values with the URL and anon key from Step 1.

JS src/supabase.js 
import { createClient } from '@supabase/supabase-js'

const SUPABASE_URL  = 'https://YOUR_PROJECT_REF.supabase.co'
const SUPABASE_ANON = 'YOUR_ANON_KEY'

export const supabase = createClient(SUPABASE_URL, SUPABASE_ANON)
5

Sign in the user anonymously

When the app loads, check whether the visitor already has a session. If not, call signInAnonymously() to create one automatically. The session is persisted in localStorage so the same user returns on the next visit.

JS src/auth.js 
import { supabase } from './supabase.js'

export async function ensureSession() {
  const { data: { session } } = await supabase.auth.getSession()

  if (session) return session   // already signed in

  // First visit — create an anonymous session
  const { data, error } = await supabase.auth.signInAnonymously()
  if (error) throw error
  return data.session
}
6

CRUD operations for todos

Because the Supabase client sends the user's JWT automatically, all queries are automatically scoped to that user by the RLS policies you created in Step 3.

JS src/todos.js 
import { supabase } from './supabase.js'

/** Fetch all todos for the current user, newest first */
export async function getTodos() {
  const { data, error } = await supabase
    .from('todos')
    .select('*')
    .order('inserted_at', { ascending: false })

  if (error) throw error
  return data
}

/** Add a new todo */
export async function addTodo(task) {
  const { data: { user } } = await supabase.auth.getUser()

  const { data, error } = await supabase
    .from('todos')
    .insert({ task, user_id: user.id })
    .select()
    .single()

  if (error) throw error
  return data
}

/** Toggle the is_done flag */
export async function toggleTodo(id, isDone) {
  const { error } = await supabase
    .from('todos')
    .update({ is_done: isDone })
    .eq('id', id)

  if (error) throw error
}

/** Delete a todo */
export async function deleteTodo(id) {
  const { error } = await supabase
    .from('todos')
    .delete()
    .eq('id', id)

  if (error) throw error
}
7

Build the UI

Wire everything together in src/main.js. Call ensureSession() first, then load todos, and attach event listeners to the form and list.

JS src/main.js 
import { ensureSession }           from './auth.js'
import { getTodos, addTodo, toggleTodo, deleteTodo } from './todos.js'

const form  = document.querySelector('#todo-form')
const input = document.querySelector('#todo-input')
const list  = document.querySelector('#todo-list')

function renderItem(todo) {
  const li = document.createElement('li')
  li.dataset.id = todo.id

  const checkbox = document.createElement('input')
  checkbox.type    = 'checkbox'
  checkbox.checked = todo.is_done
  checkbox.addEventListener('change', async () => {
    await toggleTodo(todo.id, checkbox.checked)
    span.style.textDecoration = checkbox.checked ? 'line-through' : 'none'
  })

  const span = document.createElement('span')
  span.textContent      = todo.task
  span.style.textDecoration = todo.is_done ? 'line-through' : 'none'

  const del = document.createElement('button')
  del.textContent = '✕'
  del.addEventListener('click', async () => {
    await deleteTodo(todo.id)
    li.remove()
  })

  li.append(checkbox, span, del)
  list.append(li)
}

async function init() {
  await ensureSession()

  const todos = await getTodos()
  todos.forEach(renderItem)

  form.addEventListener('submit', async (e) => {
    e.preventDefault()
    const task = input.value.trim()
    if (!task) return
    input.value = ''
    const todo = await addTodo(task)
    renderItem(todo)   // optimistically prepend
    list.prepend(list.lastChild)
  })
}

init().catch(console.error)

8

Subscribe to real-time updates (optional)

Supabase can push new rows to your client over a WebSocket, so multiple tabs stay in sync automatically. Enable it per-table in Database → Replication, then subscribe in your app:

JS src/main.js — add after init() 
supabase
  .channel('todos-changes')
  .on(
    'postgres_changes',
    { event: 'INSERT', schema: 'public', table: 'todos' },
    (payload) => renderItem(payload.new)
  )
  .subscribe()
Note: Real-time with RLS requires Supabase's row-level security for realtime feature. Make sure to enable Realtime → RLS on replication in the dashboard, or the subscription will receive no events.
9

Deploy your app

The app is plain HTML + ES modules — deploy anywhere that serves static files.

  • Vercel — push to GitHub, import the repo, done
  • Netlify — drag-and-drop your project folder
  • GitHub Pages — push to gh-pages branch
  • Cloudflare Pages — instant global CDN

Remember to set SUPABASE_URL and SUPABASE_ANON_KEY as environment variables in your hosting platform if you move them out of the source file.

Done! Every visitor now gets their own isolated todo list with zero sign-up friction — powered by Supabase anonymous auth and Row Level Security.
